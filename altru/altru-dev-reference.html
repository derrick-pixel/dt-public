🧧 Altru — Developer Architecture Reference

Singapore wedding charitable gifting platform. A guest sends a PayNow ang bao — part goes to the couple, part to an IPC charity in the couple's name. Couples get a magic-link dashboard; charities receive monthly disbursements minus a 5% platform fee. 250% IRAS tax deduction for the couple.

altru.asia Cloudflare Workers + D1 TypeScript HitPay · PayNow Resend · Twilio Stage 2 Build · May 2026
1 · Key Business Parameters
The constants that drive all business logic. Most are set in wrangler.jsonc as public env vars.
5%
Platform Fee
PLATFORM_FEE_BPS = 500. Invoiced monthly to charity on funds disbursed. Never deducted from donor payment.
14 days
Escrow Window
AUTO_REFUND_WINDOW_DAYS = 14. Couple must authorise within 14 days or gift auto-refunds via Cron job.
S$500
Large Gift Threshold
LARGE_GIFT_THRESHOLD_CENTS = 50000. Triggers a sanctions/AML check before releasing funds.
250%
IRAS Tax Deduction
All charities are IPC-registered. Donation in couple's name → 2.5× tax deduction for the couple.
~0.8%
HitPay PayNow Fee
Absorbed by Altru as COGS. Preserves the "100% to charity in gross" marketing headline.
⚠️ MAS
PSA Legal Risk

4 · API Routes
Defined in src/router.ts. Regex-matched on method + pathname. Session auth = altru_session cookie validated via HMAC + D1 lookup.
MethodPathAuthHandler (src/routes/)Description
POST /api/auth/magic-link/request Public auth.ts · postMagicLinkRequest Send magic link email to couple. Always returns {"ok":true} — no PII leak on miss.
GET /api/auth/magic-link/verify Public auth.ts · getMagicLinkVerify Consumes OTP token from email link. Sets altru_session cookie. Redirects to /couple.html.
POST /api/wedding/create Public wedding.ts · postCreateWedding Creates weddings + primary couples row. Path A (couple registers) or Path B (guest creates on behalf of couple). Queues magic link email via Resend.
GET /api/couple/me Session couple.ts · getMe Returns authenticated couple profile + wedding data. Primary data source for the dashboard UI.
POST /api/couple/verify-mobile/request Session couple.ts · postVerifyMobileRequest Sends SMS OTP to couple's mobile via Twilio. Dev mode: logs OTP to console instead.
POST /api/couple/verify-mobile/confirm Session couple.ts · postVerifyMobileConfirm Validates SMS OTP. Sets mobile_verified_at on the couples row.
POST /api/couple/set-nric Session couple.ts · postSetNric AES-encrypts NRIC using NRIC_ENCRYPTION_KEY and stores in nric_encrypted. Required for IRAS tax receipt generation.
POST /api/couple/charity-selection Session couple.ts · postCharitySelection Updates couple's featured charity and default personal split % in the wedding_charities table.
POST /api/couple/add-partner Session couple.ts · postAddPartner Creates a partner2 couples row and sends a claim-link email so the second partner can verify their identity.
GET /api/charity/list Public couple.ts · getCharityList Returns all charities with status = 'confirmed'. Used to populate charity pickers in donor wizard and couple onboarding.

6 · User Flows
Three primary journeys through the system.
🧧 Guest Sends a Gift
1
Arrives at /donor.html (via giving-aura link or homepage CTA)
2
Selects couple, enters amount (auspicious presets: $88/$168/$388/$888), adjusts charity/personal split slider — live tax calc shows IRAS deduction value
3
Chooses charity from confirmed IPC list (or defers to couple's preset)
4
Enters name, mobile, email — no account required
5
POST /api/gift/create (planned) — creates gifts row, state: pending, sets scheduled_auto_refund_at = now + 14d
6
Redirected to HitPay PayNow QR. Guest pays.
7
HitPay fires webhook → POST /api/hitpay/webhook → state: authorised
8
Guest receives confirmation email + 14-day refund link (Resend). Couple notified of new gift.
💍 Couple Registers + Authorises
1
Visit /couple.html → fill names, wedding date, email, select charity
2
POST /api/wedding/create → creates weddings + couples rows. Queues magic link email via Resend.
3
Click email link → GET /api/auth/magic-link/verify → sets altru_session cookie → redirect to /couple.html
4
Verify mobile: POST .../verify-mobile/request (Twilio SMS OTP) → POST .../confirm
5
Optional: POST /api/couple/set-nric → AES-encrypted NRIC stored for IRAS receipts
6
Optional: POST /api/couple/add-partner → partner2 receives own claim-link email
7
Share /giving-aura.html?slug=... with wedding guests
8
Dashboard (GET /api/couple/me) shows gifts in authorised state. Couple approves → state: released → disbursements queued to charity UEN + personal portion
⏱ Auto-Refund (Workers Cron)
1
Workers Cron fires on schedule (hourly). Defined in wrangler.jsonc. Calls worker.ts → scheduled().
2
Query D1: SELECT * FROM gifts WHERE state = 'authorised' AND scheduled_auto_refund_at < unixepoch()
3
For each expired gift: call HitPay refund API with payment_ref
4
Update state: authorised → auto_refunded. Write event to audit_log (actor_type: system).
5
Send refund confirmation email to guest via Resend.
⚠️ MAS PSA Risk: Altru holding funds in its own DBS account for 14 days may qualify as a "domestic money transfer service" under the Payment Services Act. Small-scale exemption (<S$3M/yr) likely applies but needs legal sign-off before launch. Lawyer brief is drafted.
7 · Gift State Machine
Every row in the gifts table transitions through these states. The state column has a D1 CHECK constraint enforcing valid values. state_changed_at is updated on every transition.
pending_claim pending authorised released auto_refunded refunded failed disputed
pending_claim→pendingGuest submits gift form; gift row created (pre-payment)
pending→authorisedHitPay webhook confirms payment success
pending→failedHitPay webhook reports payment failure
authorised→releasedCouple explicitly authorises in dashboard within 14 days
authorised→auto_refunded14 days elapsed with no couple action; Cron job fires HitPay refund
authorised→refundedGuest manually requests refund within the 14-day window
released→disputedPost-release dispute raised; manual operator action only
