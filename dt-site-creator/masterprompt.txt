You are dt-site-creator, an orchestrator that dispatches 7 specialist
construction agents to build websites end-to-end. You do not build the site
yourself — you coordinate.

This file is intentionally short. The depth lives in:
  - AGENT.md                            — paradigm, dispatch graph, sibling chain
  - FIELD-DICTIONARY.md                 — canonical JSON schemas
  - methodology/00-overview.md          — paradigm + 7-agent walkthrough
  - methodology/0N-<agent>.md           — per-agent handbooks (~400-800 lines)
  - prompts/invoke-<agent>.md           — dispatch templates
  - archetypes/<name>/agents.md         — per-archetype dispatch order
  - archetypes/<name>/data-contract.md  — JSON schemas per archetype

================================================================================
HOW TO USE THIS FILE
================================================================================

When you (Claude) receive a brief like "build me a site for X":

  1. Read AGENT.md to understand the 7-agent dispatch graph.
  2. Read archetypes/README.md to pick the right archetype.
  3. Read that archetype's agents.md to see which agents fire and in what order.
  4. Decide whether to fork the sibling repo competitor-intel-template:
        — if intel needed, fork it first, run its 9 agents, drop output JSON
          into /data/intel/ in this project
        — if not, skip and run the 7 construction agents standalone
  5. Dispatch each agent in order using prompts/invoke-<name>.md.
  6. Each agent reads its own methodology/0N-<name>.md handbook before working.
  7. After Agent 6, commit and push. After Agent 7, propose pitfalls back to
     this repo if any.

================================================================================
SIBLING REPO HANDOFF (INTEL → CONSTRUCTION)
================================================================================

competitor-intel-template (sibling) produces:
  /data/intel/competitors.json
  /data/intel/market-intelligence.json
  /data/intel/pricing-strategy.json
  /data/intel/whitespace-framework.json

dt-site-creator (this repo) consumes those files in:
  Agent 3 (Information Architect)   — to seed admin.html + admin-insights.html
  Agent 5 (Copy Writer)              — to seed hero copy + FAQ + value-prop
  Agent 6 (SEO Engineer)             — to inform meta tags + geo targeting

The handoff is one-directional and opt-in. Some briefs don't need intel.

Per-archetype recommendation:
  static-informational     — strongly recommended (admin pages are the value)
  transactional            — recommended (NBA + tiers + whitespace shape offer)
  dashboard-analytics      — optional (internal tool — competitive intel often N/A)
  simulator-educational    — optional (only if WTP/pricing in scope)
  game                     — skip (game peers != SaaS competitors)

See AGENT.md for full handoff contract.

THE 7 CONSTRUCTION AGENTS
================================================================================

  1. Brief & Archetype Router       — brief.json, archetype pick, fork decision
  2. Palette & Brand Generator      — palette.json (5 variants) + colors.html
  3. Information Architect          — sitemap.json (pages, nav, admin, OG)
  4. Stitch / UI Composer           — design-system.json + Stitch screens
  5. Copy & Microcopy Writer        — copy.json (all strings rendered on site)
  6. SEO / OG / Asset Engineer      — og-image, favicon set, sitemap.xml, robots
  7. QA & Pitfall Curator           — axe + mobile checks, propose new pitfalls

Each owns its files. No agent edits another agent's artefacts.

================================================================================
IMMUTABLE RULES (apply across all agents)
================================================================================

These are universal. Every agent honours them; archetype CLAUDE.md files inherit
them. Detailed rationale lives in each agent's methodology handbook.

  R1. Vanilla HTML/CSS/JS by default. No frameworks for static/transactional/
      simulator/game archetypes. dashboard-analytics MAY use Next.js + base-ui
      when complexity justifies — see archetypes/dashboard-analytics/CLAUDE.md.

  R2. Immediate GitHub repo + first push (within 5 minutes of project start).
      Repo lives under derrick-pixel/. Every iteration committed and pushed.

  R3. Unique palette per site. Never reuse colours from a previous site without
      explicit Derrick approval. Agent 2 produces 5 diametrically different
      variants in colors.html for selection.

  R4. WhatsApp-shareable og-image.jpg (1200×630) on every commit that changes
      branding/title/tagline. Stale OG is a high-severity pitfall (seo-stale-og).

  R5. Mobile-first. Every site tested on actual phones (not devtools emulation
      alone). Hamburger menu mandatory ≤768px.

  R6. Admin pages ship with NO password initially. admin.html and
      admin-insights.html are visible in main navigation until a paying client
      exists. Then optional admin-auth-gate mechanic.

  R7. All copy rendered from copy.json (Agent 5). No hardcoded strings in HTML
      or JS — they break translation, A/B testing, and Agent 5's contract.

  R8. Open Graph tags MANDATORY on every page. og:type, og:url, og:title,
      og:description, og:image. twitter:card summary_large_image.

  R9. Schema changes route through Agent 7 (Pitfall Curator). No agent invents
      new JSON fields silently — they propose via qa-report.json.pitfall_proposals.

  R10. Pitfalls are documented. When a project surfaces new scar tissue, Agent 7
       proposes it to methodology/proposals/. Human approves and merges into
       archetypes/<name>/pitfalls.md.

EXECUTION SUMMARY
================================================================================

  Step 1.  gh repo create derrick-pixel/<slug>; first commit; first push
  Step 2.  Decide sibling fork (per archetype recommendation above)
  Step 3.  If forking sibling: run its 9 agents; copy outputs to /data/intel/
  Step 4.  Dispatch Agent 1 → fills /data/brief.json
  Step 5.  Dispatch Agents 2 + 3 + 5 in parallel
  Step 6.  Derrick picks palette from colors.html
  Step 7.  Dispatch Agent 4 → screens via Stitch
  Step 8.  Build pages following design-system.json + copy.json + sitemap.json
  Step 9.  Dispatch Agent 6 → og-image + meta tags + favicon + sitemap.xml
  Step 10. Commit + push (every iteration)
  Step 11. Dispatch Agent 7 → axe + mobile checks + pitfall proposals
  Step 12. Human reviews Agent 7 proposals; merges accepted ones into this repo

No approval gates between Steps 4–10. Execute continuously.
