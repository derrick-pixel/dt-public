# Elitez ESOP — Internal Launch Design

**Date:** 2026-05-10
**Owner:** Derrick Teo
**Target:** Internal launch in 2 weeks
**Production URL (initial):** `esop.derrickteo.com`
**Production URL (future):** `esop.elitez.com.sg`

## Goal

Replace the current static-only ESOP platform (with its localStorage event store and shared admin code) with a real multi-user internal product. Holders log in, accept and digitally sign Letters of Offer, exercise vested options through a PayNow QR flow, and see their position in real time. Committee and Admin roles get an audit-rich oversight layer covering logins, IP addresses, and every state-changing event.

## Locked decisions (from brainstorm)

| Decision | Choice |
|---|---|
| Scope | Full platform in 2 weeks (accept + sign + exercise + QR + audit + auth) |
| Backend | Supabase (Postgres + Auth + Storage) |
| Migration of existing event store | Lift-and-shift — Supabase becomes source of truth; localStorage is a read-through cache |
| Signing approach | Typed name + checkbox + IP/UA/timestamp + doc hash, embedded in the PDF |
| Payment QR | Dynamic PayNow Corporate QR (UEN), per-exercise reference, manual admin confirmation |
| Audit log scope | Log everything (events + auth + file ops); Committee + Admin can view, with filters |
| Onboarding | Admin-triggered Supabase magic-link |
| Hosting | Cloudflare Pages, no extra CF Access perimeter |
| Architectural shape | Pure client + Supabase RLS (Approach A); one Edge Function for admin-invite |
| Legacy admin code (`elitez2026`) | Killed at launch |

## Non-goals (explicitly out of v1)

- Bank webhook / automated payment reconciliation
- Offline mutation queue (writes require connectivity)
- IP geolocation, anomaly detection, "unusual activity" flags
- 2FA / TOTP (Supabase makes it a small later add)
- Drawn-on-canvas signatures
- Mobile-app shells / push notifications
- Historical event redaction UI (events stay append-only)
- Multi-language UI

## Architecture

### System shape

```
Browser ──► esop.derrickteo.com (Cloudflare Pages, static)
                │
                │  @supabase/supabase-js (anon key)
                ▼
        ┌──────────────────────────┐
        │ Supabase project         │
        │  ├─ Auth (email+password)│
        │  ├─ Postgres + RLS       │
        │  ├─ Storage (PDFs)       │
        │  └─ Edge Function:       │
        │       admin-invite       │
        └──────────────────────────┘
```

The browser holds only the Supabase anon key. RLS policies and `SECURITY DEFINER` RPCs are the authorisation surface. Service-role key never leaves the Edge Function environment.

## Data model

### Tables

**`profiles`** — extends `auth.users`.

| Column | Type | Notes |
|---|---|---|
| id | uuid PK | FK to `auth.users.id` |
| email | text | unique, lowercase |
| full_name | text | required for signing |
| role | text | `holder` \| `committee` \| `admin` |
| holder_id | text NULL | matches `events.payload->>'holder_id'` for holder dashboards; also used by committee members who are themselves holders |
| committee_seat | text NULL | `major` \| `senior` |
| created_at | timestamptz | |
| updated_at | timestamptz | |

**`events`** — lift-and-shifted event store. Append-only.

| Column | Type | Notes |
|---|---|---|
| id | text PK | `ev_<base32>` generated by RPC |
| type | text | e.g. `grant_approved`, `letter_of_offer_signed`, `exercise_submitted`, `exercise_settled` |
| payload | jsonb | unchanged shape from current in-memory events |
| at | timestamptz | server time, set by RPC |
| actor_id | uuid | FK to `profiles.id`, NULL for `system_bootstrap` only |
| actor_role | text | role at time of event |
| client_ip | inet | from `request.headers->>'x-forwarded-for'` inside RPC |
| user_agent | text | from same |
| prev_hash | text | hash of immediately preceding event |
| hash | text | `sha256(prev_hash \|\| type \|\| at \|\| payload_json)` |

Indexes on `(at DESC)`, `(type, at DESC)`, GIN on `payload`.

**`audit_log`** — auth and file ops. Append-only. No hash chain.

| Column | Type | Notes |
|---|---|---|
| id | bigserial PK | |

| user_agent | text | |

| status | text | `draft`, `awaiting_signature`, `signed`, `voided` |

### Views

**`activity_unified`** — `UNION ALL` of `events` and `audit_log` projected into a flat shape: `at, actor_email, actor_role, action, target, ip, user_agent, metadata`. Backs the Activity Log page.

### Storage

Single private bucket: `esop-documents`. Path scheme: `{holder_id}/{kind}/{document_id}.pdf`. Access only via short-lived signed URLs from the `get_document_url(document_id)` RPC, which checks the caller is the holder or committee/admin and writes a `file_downloaded` audit row.

### RLS policies (summary)

- `profiles`: SELECT own row; committee+admin SELECT all. UPDATE own row (limited columns: full_name, email); admin UPDATE role via `update_role` RPC only.
- `events`: holders SELECT where the event is not holder-scoped (`NOT (payload ? 'holder_id')` — e.g. `valuation_added`, `window_opened`, `threshold_changed`) OR `payload->>'holder_id' = profiles.holder_id` of caller; committee+admin SELECT all. No INSERT/UPDATE/DELETE — all writes via `append_event` RPC.
- `audit_log`: SELECT for committee+admin only. No INSERT from clients (only triggers + Edge Function service role).
- `documents`: SELECT where `holder_id` matches caller; committee+admin SELECT all. INSERT/UPDATE only via RPCs.
- `payments`: SELECT where `holder_id` matches caller; committee+admin SELECT all + UPDATE status via `confirm_payment` / `cancel_payment` RPCs.

### RPCs (all `SECURITY DEFINER`)

| RPC | Caller | Effect |
|---|---|---|
| `append_event(type, payload)` | any authenticated | validates role can emit `type`, captures IP/UA, computes hash, inserts row, returns event |
| `sign_document(document_id, typed_name)` | holder owning the doc | validates name matches `profiles.full_name`, sets signature_metadata, status=signed, emits `*_signed` event |
| `finalize_signed_document(document_id, storage_path)` | the same holder | called after PDF upload; sets `documents.storage_path`, writes `file_uploaded` audit |
| `submit_exercise(grant_id, qty)` | holder | validates qty against vested/unexercised, allocates reference, creates `documents` row + `payments` row, emits `exercise_submitted` event |
| `confirm_payment(payment_id, notes)` | committee or admin | sets payment status=paid, emits `exercise_settled` event |
| `cancel_payment(payment_id, reason)` | committee or admin | sets payment status=cancelled, emits `exercise_cancelled` event |
| `get_document_url(document_id)` | holder owning doc, or committee/admin | returns 60s signed URL, writes `file_downloaded` audit |
| `update_role(profile_id, new_role)` | admin | only after a Committee resolution; logs `role_changed` |

## Auth and onboarding

### First-time onboarding

1. Admin opens "Roster" tab on `admin.html`. Sees existing roster from events. Each row has an **Invite** button.
2. Click calls `admin-invite` Edge Function with `{email, full_name, role, holder_id}`. The function authenticates the caller as admin (verifies JWT, checks `profiles.role`), uses Supabase Admin API to create the auth user, inserts `profiles` row, generates a magic-link, sends via Resend SMTP from `esop@elitez.asia`. Writes `magic_link_sent` audit row.
3. Holder receives email → clicks link → lands on `/set-password.html` → enters new password (min 12 chars; no other complexity) → redirected to `/portal.html` logged in. Writes `password_set` + `login_success` audit rows.

### Routine login

- `/index.html` form posts to `supabase.auth.signInWithPassword`.
- On success: lookup `profiles.role`, redirect: `holder → portal.html`, `committee → committee.html`, `admin → admin.html`. Writes `login_success` audit.
- On failure: Supabase Auth webhook fires → Edge Function (or Postgres trigger on `auth.audit_log_entries`) writes `login_failed` row.

### Password change and reset

- Authenticated change: avatar dropdown → "Account" → enter current password (re-auth) → new password → `supabase.auth.updateUser({password})`. Writes `password_changed`.
- Forgot password: link on login → email field → Supabase reset email → `/reset-password.html` accepts new password. Writes `password_reset_requested` then `password_changed`.

### Session model

Supabase JWTs in localStorage (default). 24h idle expiry. Tab-resume re-validates token before any write. Server-side authorisation reads `auth.uid()` and joins `profiles` for role.

### Bootstrap

Initial admin (Derrick) is bootstrapped via direct SQL once during initial setup. After that, role escalation requires a Committee resolution → `update_role` RPC.

## Holder workflows

### Acceptance and signing

Triggered by Committee approval of a grant (existing flow). System creates a `documents` row of kind `letter_of_offer`, status `awaiting_signature`. Holder portal shows a banner.

Signing modal (single screen):
1. LoO renders inline (existing `docs.js` template, now reading from Supabase).
2. `[ ] I have read and accept the terms of this Letter of Offer.`
3. Field: *Type your full legal name as it appears on your NRIC/Passport*. Validated case-insensitively against `profiles.full_name`. Mismatch → "Name doesn't match. Contact the Committee."
4. **Sign & Submit** button calls `sign_document`. RPC captures `(typed_name, ip, ua, doc_hash, at)`, sets `status=signed`, emits `letter_of_offer_signed` event.
5. Client renders the final PDF (LoO body + appended audit page) via existing html2canvas + jsPDF, uploads to Storage, calls `finalize_signed_document` to record the path.

### Exercise

Holder portal shows **Exercise** tab when an exercise window is open (existing `windows` event drives this).

1. Form: number of options to exercise (capped at vested-and-unexercised). Live calc shows SGD total.
2. **Generate Exercise Notice** button calls `submit_exercise(grant_id, qty)`. RPC validates qty server-side, allocates `EXR-YYYY-NNNNN`, generates SGQR payload encoding Elitez UEN + amount + reference, creates `payments` row (`pending`) and `documents` row, emits `exercise_submitted`.
3. Same signing modal as LoO captures the holder's intent on the Exercise Notice. PDF includes notice text + amount + reference + embedded QR + typed-name signature.
4. Portal switches to "Awaiting payment" state with the QR + bank details + reference. Holder scans, amount and reference are pre-filled by the SG banking app.

### Settlement (admin)

Admin opens **Pending payments** panel on `admin.html`. Each row: amount, reference, holder, submitted-at. Admin reconciles against the company bank statement (manual). Two buttons:
- **Mark Paid** → `confirm_payment(payment_id, notes)` → sets status=paid, emits `exercise_settled`. `calc.js` consumes this exactly as today.
- **Cancel** → `cancel_payment(payment_id, reason)` → status=cancelled, emits `exercise_cancelled`, restores option availability.

### SGQR generation

Implemented client-side from the Singapore PayNow Corporate spec (EMVCo TLV with PayNow tag set 26). Payload deterministic from `(uen, amount, reference)` so regenerating from the stored `qr_payload` always yields the same image. Tested against UOB / DBS / OCBC / Trust banking apps before launch.

## Audit and admin oversight

### What gets logged

- `events` table: every state mutation (grant_approved, allocation_changed, valuation_added, letter_of_offer_signed, exercise_submitted, exercise_settled, exercise_cancelled, resolution_proposed/voted/executed/expired, threshold_changed, roster appoint/remove, etc.).
- `audit_log` table: auth and file operations (login_success, login_failed, logout, password_set/changed/reset_requested, magic_link_sent, file_downloaded/uploaded, role_changed, audit_export_downloaded, chain_verified/chain_broken).

Both streams flow into `activity_unified` for viewing.

### Activity Log page

New tab on `admin.html` and `committee.html`. Filters at the top:
- Date range (default last 30 days).
- Actor (dropdown of profiles + free-text email contains).
- Action type (multi-select grouped: Auth · Signing · Exercise · Committee · Admin · Files).
- IP (free-text contains).
- Outcome (success / failure — relevant to logins).

Table columns: time, actor, role-at-time, action, target, IP, UA (truncated, hover for full), details (click to expand metadata jsonb).

Pagination: 100 rows per page, server-side via `.range()`.

**Download CSV** of the current filtered view. Writes `audit_export_downloaded` with the filter and row count.

### Per-holder drill-down

On the existing holder detail panel (admin), a *Recent activity for this user* section shows their last 50 rows from `activity_unified`. Same row format as Activity Log.

### Failed-login surfacing

A small badge on the topbar shows count of `login_failed` rows in the last 24h across all users. Click → Activity Log pre-filtered to failed logins. No anomaly heuristics in v1.

### Tamper-evidence

`events` keeps the prev_hash/hash chain. `audit_log` does not (would add contention without proportional benefit).

A nightly Edge Function (cron) recomputes the chain from scratch and writes a `chain_verified` audit row with `latest_hash` and `event_count`. If the chain breaks, writes `chain_broken` with the offending event id and emails admins.

### Retention

No deletion. At ~30 users on annual exercise windows + monthly committee actions, storage is trivial for years. Postgres handles it.

## Sync model (localStorage ↔ Supabase)

Source of truth is Supabase. localStorage is a read-through cache. Disagreement → Supabase wins.

### Page load

1. `auth.js` checks Supabase session; no session → `index.html`.
2. `supa.js` reads `{ events, cursor, fetched_at, cache_version }` from localStorage.
3. Fetches `events where id > cursor order by at` (paged 1000). No cache → fetches all.
4. Replays new events through `store.js` handlers exactly as today. UI renders from in-memory state.
5. Cache written back. Next load is fast.

### Mutation

Every existing emit-event call site goes through `supa.appendEvent(type, payload)`:
1. Calls `append_event` RPC. Server validates role, captures IP/UA, computes hash, returns the new row.
2. On success: pushes returned event into in-memory store, updates cache, fires existing UI re-render.
3. On failure: surfaces an error toast (`notify.js`); local state untouched. **No optimistic mutation.**

### Realtime

Supabase Realtime subscription on `events` table feeds new rows into all open tabs. Same code path as page-load replay.

### Offline behavior

Mutations fail with a clear message. Reads work from cache. No offline queue (v1.1).

### Cache versioning

Each cache entry keyed with `cache_version` constant. Bumping the constant on deploy forces full refetch.

### Bootstrap of seed data

Existing `assets/data.js` (~305 LOC of seed events) replayed once into the events table by migration `0002_seed_bootstrap.sql`, attributed to a synthetic `system_bootstrap` actor (NULL `actor_id`, `actor_role='system'`). After bootstrap completes, `data.js` is deleted from the repo.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Supabase email deliverability (magic links to corporate inboxes) | Resend SMTP from verified `esop@elitez.asia`, same as elitez-tender |
| SGQR payload correctness | Test against UOB/DBS/OCBC/Trust apps on day 7 with $0.01 reference exercises before launch |
| Hash chain genesis edge case | Explicit unit test for first event with NULL prev_hash |
| 2-week timeline assumes ~6 productive hours/day | Cuttable items if behind: Activity Log filters → CSV dump only; chain-verification cron → manual SQL check; `reset-password.html` → use Supabase hosted reset page |
| RLS policy mistake exposing other holders' data | Policy review before bootstrap; test with two seeded accounts of different holder_ids |
| Bootstrap duplicates if migration runs twice | Each seed event gets a deterministic id (`sha256(seed_index \|\| type \|\| payload)` truncated, prefixed `ev_seed_`); migration uses `INSERT ... ON CONFLICT DO NOTHING` |
| Holder cache going stale on a long-open tab missing a Realtime message | On every mutation attempt, if RPC returns `409 stale_cursor`, force a full refetch and retry once |

## Definition of done

- A net-new holder receives an invite email, sets a password, signs an unsigned LoO, and the signed PDF appears in their portal with correct signature metadata.
- A holder with vested options exercises, gets a working SGQR, and admin can mark the exercise paid; calc.js shows the new exercised position.
- Admin's Activity Log shows that holder's login (with IP), magic-link send, password set, signing event, exercise submission, and exercise settlement, with filters working.
- A failed-login attempt with the wrong password produces a `login_failed` audit row with the IP.
- Cache version bump triggers a clean refetch with no stale data.
- The hash chain verifies clean on the nightly run.
- `backend/` and `assets/data.js` are deleted; legacy `elitez2026` code path is gone.
